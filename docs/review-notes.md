# Review Notes

## 1. 검수 결론

요구 방향은 충분히 구현 가능하다.

다만 "처음부터 모든 기능을 완성한다"가 아니라 "최종 목표에는 모두 포함하되, 실패 위험이 낮은 순서로 붙인다"가 맞다.

즉, X/Twitter, Threads, YouTube, Notion, 이메일, TTS, 다중 LLM 검증을 포기하자는 뜻이 아니다. 최종 아키텍처에는 모두 포함하되, 1차 릴리즈의 성공 기준은 이메일 리포트까지로 잡는 것이 현실적이다.

## 2. 왜 단계가 필요한가

각 기능의 난이도와 실패 요인이 다르다.

- 이메일: 비교적 안정적이고 빠르게 구현 가능
- Notion: API 연동은 쉽지만 데이터베이스 스키마 설계가 필요
- TTS: 텍스트 생성보다 비용, 저장소, 재생 링크 처리가 추가됨
- X/Twitter: API 제한, 유료 정책, 계정 권한 문제가 생길 수 있음
- Threads: 공식 API 지원 범위와 접근성이 수집 요구에 맞는지 확인 필요
- YouTube: 자막 유무, 긴 영상 처리 비용, 요약 정확도 문제가 있음
- 다중 LLM: 품질은 좋아질 수 있지만 비용과 지연 시간이 증가함

## 3. 놓치면 안 되는 핵심 설계

### 3.1 Source Registry

수집 대상을 코드에 하드코딩하지 말고 설정 파일로 관리해야 한다.

각 소스는 다음 정보를 가져야 한다.

- 이름
- 타입: RSS, HTML, GitHub Release, YouTube, X, Threads
- URL 또는 API 정보
- 신뢰도
- 카테고리
- 활성화 여부
- 수집 주기

### 3.2 공통 Article 스키마

출처가 달라도 내부에서는 같은 구조로 저장해야 한다.

필수 필드:

- id
- title
- url
- sourceName
- sourceType
- publishedAt
- fetchedAt
- author
- rawText
- summary
- category
- confidenceScore
- verificationStatus

### 3.3 신뢰도 점수

소셜 미디어까지 포함하려면 모든 항목을 같은 신뢰도로 보면 안 된다.

예시:

- 공식 문서: 높음
- 공식 블로그: 높음
- GitHub Release: 높음
- 유명 엔지니어 X 계정: 중간
- 커뮤니티 게시글: 낮음
- YouTube 개인 채널: 낮음 또는 중간

신뢰도가 낮은 항목은 "확정된 사실"이 아니라 "관찰된 소식"으로 표현해야 한다.

### 3.4 사실과 해석 분리

리포트에는 다음을 분리해야 한다.

- 확인된 사실
- 사용자 업무에 대한 해석
- 아직 불확실한 점
- 추가 확인이 필요한 링크

이 구조가 없으면 AI 요약이 뉴스와 의견을 섞을 위험이 크다.

## 4. 역할 분담 판단

### 4.1 코드가 해야 하는 일

- URL 수집
- 날짜 필터링
- 중복 제거
- 스키마 검증
- 출처 신뢰도 계산
- 실패 재시도
- API 키 관리
- 이메일 발송
- Notion 저장
- 오디오 파일 저장

### 4.2 OpenAI/Codex가 잘할 일

- 기술 문서 변경 해석
- API, SDK, 코드 예제 영향 분석
- 백엔드 개발자 관점 요약
- 구현 작업 지원
- 구조화된 결과 생성

### 4.3 Claude/Anthropic이 잘할 일

- 긴 글의 맥락 요약
- 최종 리포트 가독성 개선
- 누락된 관점 검토
- 음성 브리핑 문장 다듬기

### 4.4 Gemini가 잘할 일

- Google/GCP/Gemini 관련 문맥 검토
- YouTube나 Google 생태계 자료 처리
- 필요 시 멀티모달 확장

### 4.5 LLM을 쓰면 안 되는 일

- 날짜가 수집 범위에 포함되는지 판단
- URL이 실제 접근 가능한지 판단
- 중복 여부를 전부 감으로 판단
- 공식 출처 여부를 추측
- API 키나 계정 권한 관리

이 부분은 반드시 코드와 설정으로 처리해야 한다.

## 5. MVP 권장안

사용자의 최종 목표는 전체 자동화다.

최신 개발 순서 기준의 첫 번째 제품 성공 지점은 다음으로 잡는다.

- 로컬에서 공식 출처와 일부 RSS 수집
- 코드 기반 검증
- Markdown 원문 리포트 생성
- OpenAI 요약과 Claude 리뷰
- Notion 저장
- 음성용 스크립트와 TTS
- 이메일 발송

이렇게 하면 "다중 LLM 검증", "Notion 기록", "음성 소비", "이메일 수신"을 MVP 흐름에 포함하면서도, GCP 운영과 소셜/영상 수집은 안정화 이후로 분리할 수 있다.

단, 첫 번째 하네스 작업 `001_local_collect_markdown_report`의 완료 범위는 더 좁다.

- 공식 출처 수집
- 코드 기반 검증
- 날짜 필터링
- canonical URL 중복 제거
- Markdown 원문 리포트 생성

첫 번째 하네스 작업에는 LLM, Notion, TTS, Email, GCP, X/Twitter, Threads, YouTube를 포함하지 않는다.

## 6. 대안 MVP

사용자가 처음부터 더 많은 채널을 꼭 보고 싶다면 다음처럼 제한적으로 조정할 수 있다.

- 이메일: 필수
- Notion: 필수
- 다중 LLM: 필수
- TTS: 스크립트 생성과 오디오 파일 생성
- X/Twitter: 공식 계정 또는 지정 계정만 수집
- Threads: 후순위
- YouTube: 공식 채널의 자막 있는 영상만 수집

이 방식은 최종 목표를 더 많이 포함하지만, 초기 구현 기간과 API 설정 부담이 늘어난다.

소셜과 YouTube는 제품 범위에서 제외하지 않는다. 다만 초기에는 공식 출처 기반 리포트를 먼저 안정화하고, 이후 allow-list 기반으로 확장한다.

## 7. 지금 더 정해야 할 것

1. 매일 리포트의 정확한 발송 시간
2. 가장 중요한 관심 카테고리 상위 5개
3. 반드시 추적할 공식 출처 목록
4. X/Twitter에서 추적할 계정 목록
5. YouTube에서 추적할 채널 목록
6. 이메일 발송 방식
7. Notion 데이터베이스를 새로 만들지, 기존 페이지에 붙일지
8. TTS는 어떤 음성 품질과 언어를 원하는지
9. 하루 최대 몇 개 항목까지 리포트에 포함할지

## 8. 서브 에이전트 검수 반영

서브 에이전트 검수에서 추가로 확인된 핵심 보완점은 다음과 같다.

- 사용자 여정 문서를 별도로 두는 것이 좋다.
- 기능별 수용 기준이 필요하다.
- AI 품질 정책을 분리해두는 것이 좋다.
- Source Registry와 공통 Article 스키마를 명확히 해야 한다.
- Evidence Model이 필요하다.
- Verification Gate가 필요하다.
- 비용 예산과 일일 처리 제한이 필요하다.
- 실패 정책과 보안 정책이 필요하다.

이번 문서에는 Evidence Model, Verification Gate, 비용 제한, 실패 정책, 보안 요구사항을 우선 반영했다.

## 9. 범위 판단

사용자가 원하는 최종 목표는 전체 자동화이므로 다음 기능은 제품 범위에서 제외하지 않는다.

- X/Twitter
- Threads
- YouTube
- Notion
- 이메일
- TTS
- 다중 LLM 검증

다만 구현 순서는 조절한다.

가장 현실적인 1차 제품 목표는 다음이다.

- 공식 출처 수집
- 일부 지정 소셜 계정 수집
- 코드 기반 검증
- OpenAI 요약
- Claude 리뷰
- 이메일 발송
- 음성용 스크립트 포함

이렇게 하면 사용자가 원하는 핵심 방향인 "매일 아침 검증된 리포트를 받는다"와 "다중 LLM 검증을 쓴다"를 초기부터 만족하면서, API 정책과 쿼터가 까다로운 채널은 제한적으로 시작할 수 있다.

## 10. 역할 분담 최종 판단

OpenAI/Codex는 개발자 관점의 해석과 구현에 강하므로, API 문서, 코드 변경, 백엔드 영향 분석을 맡기는 것이 좋다.

Claude/Anthropic은 긴 문서의 맥락 파악과 최종 리포트 문장 품질에 강하므로, 누락 검토와 음성 브리핑 문장 다듬기를 맡기는 것이 좋다.

Gemini는 Google/GCP/YouTube 쪽 확장에 맞다. 초기 필수 검증자는 아니지만, GCP와 YouTube 비중이 커지면 투입 가치가 있다.

규칙 기반 검증기는 날짜, URL, 중복, 공식 출처, 필수 필드, 실패 재시도를 맡아야 한다. 이 부분은 LLM보다 코드가 더 신뢰할 수 있다.
