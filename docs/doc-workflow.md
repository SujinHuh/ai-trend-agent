# Documentation Workflow

## 1. 목적

문서와 개발 과정을 계속 추적하고, 중요한 문서 변경은 서브 에이전트 검수를 거쳐 품질을 높인다.

이 프로젝트에서는 요구사항, 설계, 구현 계획, 운영 정책이 대화 중 계속 바뀔 수 있으므로 문서화 루프를 명시적으로 유지한다.

## 2. 기본 루프

1. 사용자와 요구사항 또는 설계 내용을 대화로 정리한다.
2. 관련 문서를 수정한다.
3. 작업 로그에 한 줄로 기록한다.
4. 중요한 문서 변경은 서브 에이전트로 검수한다.
5. 검수 결과에서 반영할 항목을 문서에 반영한다.
6. 반영 결과를 다시 작업 로그에 기록한다.

## 3. 서브 에이전트 검수 기준

문서와 코드의 의미 있는 변경은 서브 에이전트 검수를 기본으로 한다.

다만 토큰 사용을 줄이기 위해 작은 변경은 묶어서 검수한다.

서브 에이전트 검수가 필요한 경우:

- 요구사항 범위가 바뀐다.
- 아키텍처 또는 개발 순서가 바뀐다.
- 보안, 비용, 실패 정책이 바뀐다.
- 외부 API 또는 GCP 운영 방식이 추가된다.
- 수용 기준이나 품질 기준이 바뀐다.

간단한 오타, 문장 정리, 로그 추가는 서브 에이전트 검수 없이 진행한다.

## 4. 로그 정책

로그는 날짜별 파일에 한 줄 단위로 기록한다.

형식:

```text
HH:MM | type | summary | files
```

예시:

```text
20:45 | docs | 개발 순서를 다중 LLM 리뷰 우선으로 조정 | docs/development-plan.md
```

원칙:

- 긴 설명은 로그가 아니라 본문 문서에 쓴다.
- 로그는 무엇이 바뀌었는지 추적하는 용도다.
- 하루 단위 파일을 유지하되, 한 줄 로그로 장황해지는 것을 막는다.
- 중요한 의사결정은 `decision` 타입으로 남긴다.
- 중요한 변경은 `docs` 로그, `review` 로그, 최종 반영 로그를 구분해서 남긴다.
- `review` 로그에는 검수 결과 요약 또는 관련 문서 경로를 남긴다.
- 검수 결과 전문은 저장하지 않되, 핵심 지적 1-3개와 반영 여부는 남긴다.

## 5. 로그 타입

- `docs`: 문서 작성 또는 수정
- `decision`: 방향성 결정
- `review`: 서브 에이전트 또는 자체 검수
- `code`: 코드 작성 또는 수정
- `test`: 테스트 실행 또는 검증
- `ops`: 배포, 환경변수, GCP, 운영 설정
- `fix`: 오류 수정

## 6. 문서 변경 완료 조건

문서 변경은 다음 조건을 만족해야 완료로 본다.

- 관련 문서가 올바른 프로젝트 경로에 있다.
- 필요한 경우 서브 에이전트 검수를 받았다.
- 변경 내용이 로그에 기록됐다.
- 사용자가 다음 결정을 내릴 수 있을 정도로 문서가 명확하다.

## 7. 오류 발생 시 에스컬레이션 루프

문서, 코드, 테스트, 실행 중 큰 오류가 발생하면 즉시 에스컬레이션 루프로 전환한다.

큰 오류의 기준:

- 로컬 실행이 재현 가능하게 실패한다.
- 테스트, 타입 체크, 린트가 실패한다.
- 리포트 생성 결과가 비어 있거나 명백히 잘못됐다.
- 이메일, Notion, TTS, GCP 같은 외부 연동이 실패한다.
- API 키, 보안, 비용, 데이터 손실 가능성이 있다.
- 같은 문제가 두 번 이상 반복된다.

처리 순서:

1. `docs/logs/YYYY-MM-DD.md`에 `test` 또는 `fix` 로그를 남긴다.
2. 실패 재현 명령과 실패 증상을 확인한다.
3. 원인 후보를 좁힌다.
4. 관련 코드 또는 문서를 수정한다.
5. 같은 명령으로 재검증한다.
6. 의미 있는 오류였거나 원인이 불명확하면 서브 에이전트 검수를 실행한다.
7. 검수 핵심 지적과 반영 여부를 로그 또는 작업별 `validation_report.md`에 남긴다.

원칙:

- 실패를 조용히 넘기지 않는다.
- 재현 명령 없이 수정하지 않는다.
- 외부 연동 실패는 부분 실패 정책에 따라 가능한 결과를 유지한다.
- 보안, 비용, 데이터 손실 가능성이 있으면 작은 변경이라도 검수 대상으로 본다.
