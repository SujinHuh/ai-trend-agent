# AI Trend Agent Requirements

## 1. 프로젝트 목적

AI Trend Agent는 AI 및 백엔드 기술 변화 정보를 매일 자동 수집하고, 검증된 요약 리포트로 만들어 사용자에게 전달하는 개인용 자동화 시스템이다.

목표는 사용자가 매일 여러 소스에서 흩어진 정보를 직접 찾지 않아도, 출근 시간에 읽거나 들으면서 핵심 변화를 파악할 수 있게 하는 것이다.

## 2. 핵심 목표

- 매일 오전 7시 전후 최신 기술 리포트를 자동 생성한다.
- 공식 출처, 소셜 미디어, 영상 자료를 모두 수집 대상으로 설계한다.
- 날짜, URL, 중복, 출처 신뢰도를 코드로 1차 검증한다.
- LLM은 요약 품질, 맥락, 누락, 실무 영향도 검토에 사용한다.
- 이메일, Notion, 음성 소비 채널을 모두 최종 목표에 포함한다.
- 초기 구현은 작게 시작하되, 최종 구조에서 X/Twitter, Threads, YouTube, TTS, 다중 LLM 검증을 제외하지 않는다.

## 3. 사용자 시나리오

### 3.1 매일 아침 리포트

사용자는 아침 7시쯤 이메일을 받는다.

메일 상단에는 오늘 가장 중요한 변화 3-5개가 표시된다.

각 항목에는 다음이 포함된다.

- 한 줄 요약
- 왜 중요한지
- 백엔드 또는 AI 업무에 미치는 영향
- 원문 링크
- 검증 상태
- 더 읽을지 여부를 판단할 수 있는 상세 요약

### 3.2 출근 중 음성 소비

사용자는 핸드폰으로 음성 요약을 듣는다.

음성 콘텐츠는 문서 요약을 그대로 읽는 것이 아니라, 이해하기 쉬운 구어체 브리핑이어야 한다.

### 3.3 나중에 다시 보기

리포트는 Notion에도 저장된다.

사용자는 Notion에서 날짜, 카테고리, 중요도, 출처, 읽음 여부로 자료를 다시 찾을 수 있다.

- 나중에 읽을 자료

## 6. 검증 구조

### 6.1 코드 기반 검증

기계적으로 확인 가능한 것은 LLM이 아니라 코드가 검증한다.

- URL 접근 가능 여부
- 게시일 또는 업데이트 날짜
- 수집 기간에 포함되는지
- 중복 URL 또는 중복 주제 여부
- 공식 출처 여부
- 원문 본문 추출 성공 여부
- 필수 메타데이터 존재 여부

### 6.2 LLM 기반 검증

LLM은 의미와 품질을 검토한다.

- 요약이 원문에 근거하는지
- 과장된 표현이 있는지
- 중요한 맥락이 빠졌는지
- 사용자의 업무와 어떤 관련이 있는지
- 불확실한 내용을 사실처럼 말하고 있지 않은지
- 더 확인해야 할 원문이 있는지

## 7. 다중 LLM 역할 분담

### 7.1 OpenAI 또는 Codex 계열

주요 역할:

- 개발자 문서, API 변경, 코드 예제, SDK 변경 분석
- 백엔드 개발자 관점의 실무 영향도 정리
- 자동화 파이프라인 코드 작성 및 유지보수
- 수집 결과를 구조화된 JSON으로 변환

강점:

- 코드와 개발 워크플로우 해석
- API 문서 변화 파악
- 자동화 구현과 테스트 설계

주의점:

- 최신 사실은 반드시 원문과 날짜로 검증해야 한다.

### 7.2 Claude 또는 Anthropic 계열

주요 역할:

- 긴 문서, 정책, 블로그 글의 맥락 요약
- 누락된 관점 검토
- 최종 리포트 문장 품질 개선
- 음성 브리핑용 자연스러운 구어체 작성

강점:

- 긴 텍스트 처리
- 문맥 유지
- 사용자에게 읽히기 쉬운 설명 생성

주의점:

- 기술 세부 버전, 요금제, 제품명은 원문 링크로 재확인해야 한다.

### 7.3 Gemini 또는 Google 계열

주요 역할:

- GCP, Google AI, Gemini 관련 자료 검토
- YouTube 자막 기반 요약
- Google 생태계 문서 이해

강점:

- Google Cloud와 Google 제품군 맥락
- 멀티모달 또는 영상 관련 확장 가능성

주의점:

- 프로젝트 초기에는 필수 구성요소가 아니라 확장 검증자로 둔다.

### 7.4 규칙 기반 검증기

주요 역할:

- 날짜, URL, 중복, 출처, 스키마 유효성 검증
- LLM 결과가 필수 필드를 채웠는지 확인
- 신뢰도 점수 산정

강점:

- 비용이 낮다.
- 반복 가능하다.
- LLM보다 결정적이다.

결론:

다중 LLM은 "서로 맞다고 말하게 하는 구조"가 아니라, 각자 강점이 있는 검토 역할을 나누는 방식으로 사용한다.

## 8. 전달 채널

이 섹션은 전체 제품 기준이다. Task 001에는 이메일, Notion, TTS 전달 채널을 포함하지 않는다.

### 8.1 이메일

MVP의 첫 번째 전달 채널이다.

필수 항목:

- 오늘의 핵심 3-5개
- 카테고리별 요약
- 원문 링크
- 상세 리포트 링크
- 음성 스크립트 또는 오디오 링크

### 8.2 Notion

리포트 아카이브와 검색용 저장소다.

필수 속성:

- 날짜
- 제목
- 카테고리
- 중요도
- 출처
- 원문 링크
- 검증 상태
- 읽음 여부
- 상세 요약

### 8.3 TTS

출근 중 듣기 위한 음성 콘텐츠다.

초기에는 음성용 스크립트를 먼저 생성하고, 이후 TTS API로 오디오 파일을 생성한다.

스크립트 기준:

- 5-10분 분량
- 구어체
- 링크를 읽지 않음
- 숫자, 버전, 제품명은 또렷하게 설명
- 핵심 변화와 실무 영향 중심

## 9. 시스템 아키텍처 초안

### 9.1 GCP 기반 구성

- Cloud Scheduler: 매일 실행 트리거
- Cloud Run: 수집, 요약, 검증, 발송 작업 실행
- Firestore 또는 Cloud SQL: 수집 항목과 리포트 저장
- Cloud Storage: 원문 스냅샷, 오디오 파일 저장
- Secret Manager: API 키 관리
- Cloud Logging: 실행 로그와 실패 원인 기록

운영 시간 기준:

- Cloud Scheduler의 schedule timezone은 반드시 `Asia/Seoul`로 설정한다.
- GCP 기본 타임존인 UTC에 의존하지 않는다.
- 애플리케이션 내부의 리포트 날짜 계산도 `Asia/Seoul`을 기준으로 한다.
- 발송 목표 시각은 매일 `07:00 KST`다.
- 수집 윈도우 종료 시각은 `06:50 KST`로 두고, `07:00 KST`까지 10분 가공 버퍼를 확보한다.

### 9.2 파이프라인

1. Source Registry에서 수집 대상 목록을 읽는다.
2. 각 소스별 Collector가 신규 자료를 수집한다.
3. Normalizer가 공통 스키마로 변환한다.
4. Rule Verifier가 URL, 출처, 필수 필드를 검증한다.
5. Date Filter가 수집 윈도우 기준으로 항목을 분류한다.
6. Deduplicator가 canonical URL 기준으로 중복 항목을 제거한다.
7. Ranker가 중요도를 계산한다.
8. LLM Summarizer가 요약을 생성한다.
9. LLM Reviewer가 요약의 근거와 누락을 검토한다.
10. Final Editor가 최종 리포트와 음성용 스크립트를 만든다.
11. Notion Writer가 최종 리포트와 항목을 저장한다.
12. TTS Worker가 음성 파일을 생성하고 저장소에 업로드한다.
13. Email Delivery가 핵심 요약, Notion 링크, 음성 스크립트 또는 오디오 링크를 발송한다.

- 중복 제거
- 날짜 필터링

### Phase 2: 로컬 LLM 요약 및 다중 LLM 리뷰

- OpenAI 요약

- 중요 항목에 대한 추가 검증
- 검증 결과와 불확실성 표시

### Phase 3: Notion 저장

- 일일 리포트 Notion 저장

### Phase 4: 음성 브리핑

- 음성용 스크립트 생성

### Phase 5: 이메일 발송

- 핵심 요약 발송
- Notion 링크 포함

### Phase 6: GCP 배포

- Cloud Run 배포
- Cloud Scheduler 자동 실행

### Phase 7: 소셜 및 영상 확장

- X/Twitter 수집
- Threads 수집
- YouTube 자막 수집 및 요약
- 소셜 출처는 신뢰도 점수를 낮게 시작하고 공식 출처와 교차 확인

## 11. 성공 기준

- 매일 오전 7시 전후 리포트가 생성된다.
- 리포트 항목은 원문 링크를 가진다.
- 게시일이 오래된 자료를 최신 소식처럼 전달하지 않는다.
- 사용자는 10분 안에 핵심 내용을 파악할 수 있다.
- 음성 콘텐츠만 들어도 주요 흐름을 이해할 수 있다.
- 실패 시 원인과 실패 단계가 기록된다.
- LLM 요약은 원문 근거를 포함한다.

## 12. Evidence Model

각 요약 항목은 반드시 근거 정보를 가진다.

필수 근거 필드:

- sourceUrl
- sourceName
- sourceType
- publishedAt
- fetchedAt
- evidenceExcerpt
- confidenceScore
- verificationStatus

요약 문장은 가능한 한 원문 근거와 연결되어야 한다.

근거가 부족한 항목은 메인 리포트가 아니라 "확인 필요" 섹션으로 이동한다.

## 13. Verification Gate

다음 조건에 해당하는 항목은 최종 리포트 상단에 올리지 않는다.

- 원문 URL이 없다.
- URL 접근에 실패했다.
- 게시일 또는 업데이트 날짜를 확인할 수 없다.
- 수집 기간 밖의 오래된 자료다.
- 공식 출처 여부를 확인할 수 없는데 확정 표현을 사용한다.
- 모델명, 가격, 버전 번호, 사용량 제한이 원문과 불일치한다.
- LLM 리뷰에서 "근거 부족" 또는 "추측 가능성 높음"으로 표시됐다.

이 항목들은 삭제하지 않고 "확인 필요" 또는 "낮은 신뢰도 소식"으로 따로 저장한다.

## 14. 비용 및 처리 제한

초기 운영 비용을 통제하기 위해 하루 처리량 제한을 둔다.

초기 기본값:

- 하루 최대 수집 후보: 200개
- LLM 요약 대상: 랭킹 상위 30개
- 다중 LLM 검증 대상: 중요도 상위 10개
- 이메일 메인 리포트: 10개 내외
- 음성 브리핑: 5-10분 분량
- YouTube: 지정 채널의 최근 영상 중 자막 있는 영상만 처리
- X/Twitter: 지정 계정 또는 키워드 allow-list만 처리
- Threads: 초기에는 allow-list 기반으로 제한

정책, 가격, 모델, 보안, 사용량 제한 변경 항목은 우선순위를 높인다.

## 15. 실패 정책

부분 실패가 있어도 가능한 결과는 전달한다.

- Notion 저장 실패: 이메일은 발송한다.
- TTS 생성 실패: 음성용 스크립트만 이메일에 포함한다.
- 소셜 수집 실패: 공식 출처 리포트만 생성한다.
- 특정 LLM 실패: 사용 가능한 LLM 결과와 실패 표시를 함께 남긴다.
- 이메일 발송 실패: 로그를 남기고 재시도한다.
- 전체 파이프라인 실패: 실패 원인과 실패 단계를 알림으로 남긴다.

실패는 조용히 무시하지 않는다.

## 16. 보안 요구사항

API 키와 개인 인증 정보는 코드와 문서에 직접 저장하지 않는다.

필수 원칙:

- GCP Secret Manager 사용
- 최소 권한 OAuth scope 사용
- 로그에 API 키, 개인 이메일, 인증 토큰 출력 금지
- Notion, Gmail, X, YouTube, OpenAI, Anthropic, Gemini 토큰 분리 관리
- 토큰 회전 가능하도록 설계
- 외부 API 약관과 rate limit 준수
