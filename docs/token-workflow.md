# Token Workflow

## 1. 목적

Codex와 협업할 때 토큰 사용을 줄이면서도 문서화, 검수, 구현 품질을 유지한다.

## 2. 기본 원칙

- 큰 문서를 매번 다시 읽지 않고 필요한 섹션만 확인한다.
- 변경은 관련 파일에만 작게 적용한다.
- 로그는 한 줄로 남긴다.
- 서브 에이전트 검수는 의미 있는 변경을 묶어서 실행한다.
- 검수 결과는 전체 전문을 문서에 복사하지 않고 핵심 반영 사항만 기록한다.
- 검수 근거가 사라지지 않도록 핵심 지적 1-3개와 반영 여부는 로그나 작업 검증 문서에 남긴다.

## 3. 작업 시작 방식

새 작업을 시작할 때는 다음 순서로 확인한다.

1. `docs/logs/YYYY-MM-DD.md`의 최근 로그
2. 관련 작업 문서
3. 관련 요구사항 문서의 필요한 섹션
4. 코드 변경 대상 파일

전체 문서를 모두 다시 읽는 것은 요구사항이 크게 바뀌었을 때만 한다.

## 4. 서브 에이전트 사용 방식

서브 에이전트는 계속 사용하는 것을 기본으로 하되, 토큰 절약을 위해 단위를 조절한다.

사용 기준:

- 요구사항, 아키텍처, 개발 순서, 보안, 비용, 운영 정책 변경은 검수한다.
- 코드 구현 후 주요 기능 단위로 검수한다.
- 단순 로그 추가, 오타 수정, 파일 이동 확인은 검수하지 않는다.
- 여러 작은 문서 변경은 하나의 검수 요청으로 묶는다.

## 5. 로그 방식

로그는 날짜별 파일에 한 줄로 남긴다.

형식:

```text
HH:MM | type | summary | files
```

로그에는 상세 설명을 쓰지 않는다.

상세한 이유, 결정 배경, 대안 비교는 해당 문서 본문에 쓴다.

서브 에이전트 검수 결과는 전체 전문 대신 다음만 남긴다.

```text
HH:MM | review | 핵심 지적: 문서 간 순서 불일치, 반영 완료 | docs/requirements.md
```

## 6. 대화 방식

토큰을 줄이기 위해 Codex는 다음 방식으로 응답한다.

- 완료된 변경과 다음 판단만 짧게 말한다.
- 긴 설명이 필요한 내용은 문서에 쓰고 답변에서는 파일 링크만 제공한다.
- 이미 합의된 내용은 반복 설명하지 않는다.
- 사용자가 요청하지 않은 전체 문서 요약은 하지 않는다.
